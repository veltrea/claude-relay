# claude-relay: простое сохранение памяти сессий Claude Code

## Предыстория

Я форкнул [claude-mem](https://github.com/anthropics/claude-mem) (плагин памяти сессий для Claude Code), чтобы попробовать запустить его с локальными LLM. Почитал исходники и, честно говоря, пользоваться этим было невозможно.

Архитектура, которая шлёт запрос на AI-сжатие при каждом вызове инструмента, fetch без таймаутов, отсутствие стратегии повторных попыток, путаница между liveness и readiness, необратимая обработка, которая выбрасывает сырые данные после сжатия -- реализация, в которой не соблюдены базовые принципы computer science. Подробнее я написал в [отдельной статье](https://note.com/veltrea/n/n791d1defada0).

Если использовать Claude API, проблемы не всплывают на поверхность. Но стоит переключиться на локальную LLM -- и всё становится критичным. Я пытался исправить форк, но проблема в самом подходе к проектированию, точечными патчами это не решается.

А потом я подумал: а зачем вообще сжимать через AI? Claude Code и так записывает все данные сессий в JSONL в `~/.claude/projects/`. Достаточно загнать это в SQLite, а при поиске дать самому Claude разобраться в сырых данных с его контекстом в 1M токенов. Никакого AI-сжатия, никаких демонов.

Так появился **claude-relay**, написанный с нуля.

## Что это такое

- Один бинарник на Rust (около 1 600 строк)
- Подключается к Claude Code как MCP-сервер и предоставляет инструменты для поиска по прошлым сессиям
- Без демонов. Инкрементальный импорт JSONL происходит при старте сессии и при вызовах инструментов
- Можно архивировать старые данные в Markdown и удалять их из SQLite

## Установка

Нужна среда сборки Rust.

```bash
git clone https://github.com/veltrea/claude-relay.git
cd claude-relay
cargo build --release

# Register as MCP in Claude Code
claude mcp add --transport stdio --scope user claude-relay -- $(pwd)/target/release/claude-relay serve
```

Если хотите добавить в PATH, скопируйте `target/release/claude-relay` куда удобно.

## Использование

### Сначала импортируем JSONL

```bash
claude-relay ingest ~/.claude/projects/
claude-relay ingest path/to/session.jsonl
claude-relay db stats
```

В моём окружении получилось около 48 сессий и 75 000 записей.

### Использование через Claude Code

Инструмент зарегистрирован как MCP, так что можно просто спрашивать внутри сессии Claude Code.

- "What did I work on yesterday?"
- "Find that OAuth fix"
- "What happened between March 20-23?"
- "Show me recent sessions"

Под капотом вызываются MCP-инструменты: `memory_search`, `memory_list_sessions`, `memory_get_session` и другие.

### Можно использовать и через CLI

Есть команды для управления прямо из терминала. Поскольку вызовы через MCP-инструменты расходуют токены, административные задачи лучше делать через CLI.

```bash
claude-relay list
claude-relay list --date 2026-03-23
claude-relay export <session_id>
claude-relay export --date 2026-03-23
claude-relay db reset
claude-relay query "SELECT type, COUNT(*) FROM raw_entries GROUP BY type"
claude-relay write "test message" --type user
```

## Про архитектуру

### Сохраняем всё, фильтруем при чтении

Сначала я хотел сохранять только `user` и `assistant`, но потом подумал: «А почему бы не загнать всё и фильтровать через WHERE при чтении?» Так что `system`, `progress`, `queue-operation` -- всё попадает в базу. Если потом захочется посмотреть данные, которые раньше казались ненужными, они будут на месте.

### Без демонов

Рассматривал вариант с резидентным демоном для мониторинга файлов (типа chokidar), но отказался. Инкрементальный импорт происходит в хуке SessionStart и при вызовах MCP-инструментов. Для каждого JSONL запоминается байтовое смещение «до куда прочитали», и обрабатываются только новые строки.

### Архивирование

В файле конфигурации (`~/.claude-relay/config.json`) можно указать `retention_days`, чтобы экспортировать устаревшие данные в Markdown и удалить их из БД. По умолчанию -- 30 дней.

```jsonc
{
  "retention_days": 30,
  "archive_dir": "~/.claude-relay/archive"
}
```
