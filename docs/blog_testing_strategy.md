# 個人開発のテスト戦略 ── 「会話ログ検索ツール」を4層テストで守った話

## はじめに

私はふだん、ひとりでソフトウェアを作っています。

今回つくったのは claude-relay という、Claude Code（Anthropic社のAIコーディングアシスタント）専用の会話ログ検索ツールです。Rustで書いたシングルバイナリで、Claudeとのやりとりを片っ端からSQLiteに入れて全文検索できるようにするものです。

なぜこんなものが必要かというと、Claudeにはセッションをまたいだ記憶がないからです。ChatGPTなど一部のAIは過去の会話をある程度覚えていてくれますが、Claudeは新しいセッションを開くたびにまっさらな状態に戻ります。本来はセッションを閉じる前にハンドオーバーメモを書かせたり、CLAUDE.mdに要点をまとめさせたりするのですが、これが結構抜け落ちます。さらに厄介なのが、セッションのコンテキスト上限に達してしまったケースです。こうなるとClaudeはもう新しい指示を受け付けられないので、「閉じる前にメモを書いて」というお願い自体ができません。作業の記録がどこにも残らないまま、セッションが終わってしまう。「あれ、先週やったあの修正の詳細なんだっけ」と思ったとき、どこにも残っていない──そういう体験が出発点でした。

私はどちらかというとテストから先に組むタイプです。その理由の一つは、バージョンアップのたびにちょっとしたミスで問題が混入するからです。機能追加やリファクタリングをするたびに、以前到達してた動作が静かに壊れる。手動で確認するのは限界があるので、自動化して毎回の検証を仼わせる必要がある。今回のツールは処理のステップが多く、どこか一箇所でも壊れると全体が動かなくなる構造だったので、最初から4層に分けてテストを組みました。

この記事では、私がどんなテストをどういう順番で組んだのか、なぜそれが必要だったのか、ということを書いていきます。偉そうなことを言うつもりはなく、「こうしたらデバッグの時間がだいぶ減りましたよ」という体験記です。

---

## そもそも何をテストしたいのか

このツールには、ざっくり4つの段階があります。

```
AIと会話 → ログが記録される → DBに取り込む → 検索で見つかる
```

1. AIと会話したら、その内容がファイルに書き出される（JSOL記録）
2. そのファイルをSQLiteデータベースに取り込む（この処理を ingest と呼んでいます）
3. データベースから全文検索ができる（search）
4. 上の1〜3を通しで流したとき、最初のフレーズが最後にちゃんと見つかる（E2Eリコール）

ポイントは、この4段階のどこが壊れても最終結果は「検索で出てこない」になる ということです。E2Eテスト（端から端まで通すテスト）だけだと、「出てこない」のはわかっても「どこが壊れたのか」がわかりません。そこで各段階にテストを置くことにしました。

---

## Layer 1: AIとの会話がファイルに記録されるか

### これは何？

AIに「この文字列を覚えて」と送って、その文字列がログファイルに書き出されているかを確認するテストです。

### なぜ必要？

「AIに話しかけたのに、そもそもログが残っていなかった」──実はこれ、意外と起こります。CLIの起動オプションが変わった、作業ディレクトリが変わった、AIがエラーを返してログ自体が生成されなかった、など原因はいろいろです。

このテストがないと、データベースに何も入っていないときに「取り込み処理のバグかな？ いやDBの問題かも」と延々迷うことになります。

### どうやっている？

```
① ランダムな文字列を生成（例: LAYER1_TEST_0001_a3f8c2b1）
② AIにワンショットで送信
③ ログファイル群の中にその文字列があるか grep で確認
④ ついでに、ログの各行が壊れたJSONになっていないかもチェック
```

ランダム文字列を使うのがコツです。「テスト用のメッセージ」みたいな曖昧なものだと、過去のログにもマッチしてしまって偽陽性になりかねません。毎回ユニークな文字列を作ることで「本当に今回の会話がこのファイルに記録されたか」を正確に判定できます。

---

## Layer 2: ファイルからデータベースに正しく取り込めるか

### これは何？

テスト用のログファイルを自分で作って、取り込みコマンド（`ingest`）に渡し、データベースに正しく入ったかをSQLで直接確認するテストです。

### なぜ必要？

この取り込み処理はツールの心臓部です。ここでは以下のことを全部検証しています。

- 基本動作: ファイルの3行が3件のレコードになるか
- 冪等性: 同じファイルを2回取り込んだとき、データが重複しないか
- 差分追加: ファイルに行を追記したとき、新しい行だけが取り込まれるか
- 壊れた行のハンドリング: JSONとして不正な行があったとき、その行だけスキップして後続を処理できるか
- データ型: user / assistant / system の区別がDBに正しく入っているか

### 「2回やっても同じ結果」のテストは地味に大事

ツールの運用を想像してください。ユーザーは毎日何度も取り込みを実行します。もしデータが重複して入ってしまうと、検索結果に同じ内容が10件並ぶようなことになります。「同じ操作を何回繰り返しても結果が変わらない」──専門用語では冪等性（べきとうせい）と言いますが、要するに「うっかり2回実行しちゃっても大丈夫」を保証するテストです。地味ですが、これがないと実際に使うときに困ります。

### 壊れたJSONの処理

実際のログファイルは、稀に壊れることがあります。プロセスが途中で終了した、ディスクが満杯になった、など。1行壊れたからといって数百行の正常データを全部無視するわけにはいきません。テストでは「壊れた行の前後のデータが正しく取り込まれること」を確認しています。

---

## Layer 3: データベースから正しく検索できるか

### これは何？

テストデータをDBに入れた状態で、検索コマンドを実行し、期待通りのデータが返ってくるかを確認するテストです。

### なぜ必要？

データベースに入っていても、検索で見つからなかったら意味がありません。SQLiteのFTS5（全文検索エンジン）は便利ですが、インデックスの張り方やクエリの書き方を間違えると「データはあるのにヒットしない」状態になります。

### テスト項目

- 完全一致: ユニークなフレーズで検索して見つかるか
- 存在しないフレーズ: でたらめな文字列で検索したとき、ちゃんと0件になるか
- 日本語: 日本語を含むフレーズが検索できるか（FTS5の日本語対応は自明ではありません）
- limit: 「最大N件」の制約が効くか
- キーワード検索: 文中のキーワードで検索して、関連エントリが出てくるか
- セッション一覧: 特定のセッションIDが一覧に出るか

### 日本語検索のハマりどころ

全文検索エンジンは英語圏で生まれたものが多く、日本語は「単語の区切り」がスペースで分かれていないという根本的な違いがあります。FTS5のデフォルト設定だと日本語がまったくヒットしない、ということもあり得るので、ここを明示的にテストしておくことには意味があります。

---

## Layer 4: E2Eリコールテスト（本丸）

### これは何？

Layer 1〜3を全部つなげた「端から端まで」テストです。AIに30個のフレーズを実際に送信し、DBへの取り込みを走らせ、検索で全部見つかるかを確認します。

### なぜ30件？

1〜2件だと「たまたまうまくいった」の可能性を排除できません。30件あれば、タイミングの問題やレースコンディションの問題が確率的に出やすくなります。逆に100件だと実行時間がかかりすぎて日常的に回せません。30件は「信頼性とスピードのバランス」で選んだ数です。

### 2つのモード

このテストには2つのモードがあります。

ワンショットモード

```bash
bash tests/memory_recall_test.sh oneshot 30
```

AIに `claude -p`（プログラム呼び出し）で1件ずつ送信します。1回の呼び出しで1つのフレーズを送り、終わったら次──という逐次処理です。安定性が高く、CI環境でも回せます。

インタラクティブモード

```bash
bash tests/memory_recall_test.sh interactive 10
```

tmux（ターミナルマルチプレクサ）を使って、AIのインタラクティブセッションを立ち上げ、実際にユーザーが手で打つのと同じ形で送信します。起動に20秒、応答待ちに15秒という待ち時間が入るので遅いですが、ワンショットでは見つからない「セッション管理のバグ」を発見できます。

### 最新の結果

```
Total phrases:  30
Seed OK:        30
Seed FAIL:      0
Found:          30
Not Found:      0
Recall Rate:    100.0%
Verdict:        PASS — 100% recall
```

30フレーズすべてが送信に成功し、すべてが検索で見つかりました。

## テストの積み重ねが効く場面

ここまで読んで「そこまでやる必要ある？」と感じた方もいるかもしれません。私自身は、身内にエンジニアがいた影響で「コードを書いたらまずテストを書く」が子どもの頃からの習慣になっています。「テストファースト」なんて言葉が世に出るよりずっと前に、そういうもんだと叩き込まれた感じです。なので「なぜテストを書くのか」をあらためて考えたことはあまりないのですが、具体例で示すとこういうことです。

### 例: 「検索で何も出てこない」問題

ある日、memory_searchで何もヒットしなくなりました。もしE2Eテストしかなかったら、こうなります。

```
→ E2Eが失敗している
→ AIの呼び出しが問題？ ログの書き出しが問題？
   取り込みが問題？ 検索が問題？
→ 全部を手で確認していく……
```

でもLayer 1〜3のテストがあると、切り分けが一瞬です。

```
→ Layer 1: PASS → ログは書けている
→ Layer 2: PASS → 取り込みも正常
→ Layer 3: FAIL → 検索まわりが原因
→ FTS5のインデックス再構築で解決
```

10分で終わりました。テストを書くのに使った時間は、このたった1回のデバッグですでに回収した感覚があります。

## AIとテストの相性

このプロジェクトはAIコーディングアシスタント（Claude Code）と一緒に開発しました。AIとテストの組み合わせで面白いのは、AIが「テスト関数を実行する」だけでなく、失敗の内容を見て原因を推定し、修正を試みてまた実行するという動きを自律的にやってくれる点です。うまくはまると、PDCA を勝手に回してくれます。

今回は元々バグがほとんど出なかったので、その「自律的なデバッグループ」が劇的に回る場面はありませんでした。ただ、網羅的なテストを人力でやるのはそれなりに疲れる作業で、AIにテスト実行と結果確認を任せることで集中力を別のところに使えたのは事実です。

もう一つ、意外な恩恵がありました。AIにテストの仕組みを考えさせる過程で、設計上の問題を指摘してくれることがあるのです。「このクエリは件数が増えると遅くなりそう」「差分取り込みのロジックに穴がある」「同じ操作を繰り返したとき結果が変わってしまう」──こういった、自分ひとりだと見落としがちな問題を、テストを考えさせるときに気づいてくれることがあります。私はなるべく積極的にこれを活用するようにしています。

AIとのペア開発では、テストが「AIへの仕様伝達手段」になるという側面もあります。「このAPIはこういう入力にこういう結果を返すべき」という意図がテストに書かれていれば、AIが実装を変更するときに誤った方向へ進みにくくなります。テストを書く理由がまた一つ増えた気がしています。

## 個人開発でテストを書くコツ

最後に、私が気をつけていることをいくつか書いておきます。

### ランダムフレーズを使う

テストデータは毎回ランダムに生成します。`openssl rand -hex 4` で8文字のランダム文字列をつけるだけで、「前回のテストの残りカスにマッチしてしまった」系の偽陽性をほぼ消せます。

### 壊れたデータを意図的に入れる

正常系だけテストしても不十分です。壊れたJSONの行、空のファイル、存在しないディレクトリ──こういうのを意図的にテストに含めておくと、本番で慌てなくて済みます。

### テスト結果をファイルに残す

テストスクリプトは結果を `/tmp/` 以下にタイムスタンプ付きで保存するようにしています。「先週は通ってたのに今週はダメ」というとき、過去の結果を見返せると原因の絞り込みが早くなります。

### コマンド一発で完結するようにする

テストの実行に手作業が必要だと、だんだん億劫になって回さなくなります。`bash tests/test_layer2_ingest.sh` のようにコマンド一発で完結するようにしておくと、コードを変えたあとに「とりあえず回しとくか」という気持ちになれます。こうしておけば、将来的にCI（コードを変更するたびにテストを自動実行してくれる仕組み）を導入するときもそのまま乗せられます。

## おわりに

テストについて「書くのが面倒」「意味があるのか不安」という声をよく聞きますが、正直なところ私にはその感覚がよくわかりません。「AIにこの関数のテストコード書いて、条件はこれ」と伝えれば一分もかからない。仕様書を書くときに期待値を書かない人はいないはずで、テストというのはその期待値を実行可能な形にしたものに過ぎないと思っています。

ただ一度バグに遭遇すると話は変わって、テストがある場所は30秒で原因がわかり、ない場所は30分かかる──この差を体験すると、もう戻れなくなるはずです。

バイブコーディングの時代になって、コードを書くスピードは誰でも上がりました。だからこそ、設計の質とテストの有無が、より大きな差を生む時代になってきていると感じています。

もう一つ現実的な話をすると、AIエージェントには「大胆に書き換えてバグを新しく作る」という傾向があります。特に Antigravity はこれで有名で、コストの問題でClaudeをスポット的にしか使えない環境だとAntigravityに頼る場面が増えますが、文脈を十分に把握しないまま大規模な変更を加えてくることがある。テストがなければ、そのバグは気づかないままリリースされます。テストはAIエージェントへの安全装置でもある、というのが最近の実感です。

個人開発だからこそ、テストで「未来の自分」と「AIエージェントの暴走」の両方から守ってあげる。

逆に言えば、現時点ではテストファーストな開発スタイルを持っている人にとって、大胆に書き換えるAIエージェントはコスパ最強の組み合わせになりえます。テストというセーフティーネットがある分、AIが大胆に動いても安全に止められる。もっとも、AIモデルは急速に進化していて、次のバージョンではこういった傾向が改善されていく可能性もあります。それでも「AIの個性を理解して使う」という姿勢は、どの時代でも有効だと思っています。
